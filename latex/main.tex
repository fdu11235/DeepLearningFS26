\chapter*{Management Summary}
\thispagestyle{frontmatter}
\addcontentsline{toc}{chapter}{\textbf{Management Summary}}

Credit-card fraud is rare relative to legitimate spending — well under one percent of transactions in the public Sparkov-simulator dataset are fraudulent — yet a single missed fraud event imposes asymmetrically high costs on banks and cardholders. Our previous classical machine-learning study of this dataset reached a fraud-class F\(_1\) of $0.48$ (precision $0.59$, recall $0.41$) using a tuned Random Forest, with SMOTE oversampling actively degrading test-set generalisation. The present paper asks whether a sequence-based deep learning model that conditions each prediction on a credit card's recent transaction history can close that gap, and whether explicit class-imbalance handling adds value once the model is allowed to use such context.

We train a two-layer LSTM over per-card sliding windows of length 20, with right-padded \texttt{pack\_padded\_sequence} so the recurrent cell only updates on real timesteps and with cross-split history prepended at validation/test boundaries. We compare three imbalance-handling variants: plain binary cross-entropy on the imbalanced data, cost-sensitive cross-entropy with $\text{pos\_weight} \approx 172$, and sequence-level SMOTE. The plain-BCE LSTM achieves PR-AUC $0.969$ on the held-out test set — versus $0.509$ for the Random Forest baseline — and a fraud-class F\(_1\) of $0.94$ at the default threshold, with only $52$ false positives across $553{,}574$ legitimate transactions. Both imbalance-handling variants underperform the plain configuration, contradicting a default of the imbalanced-classification literature. The headline gain is structural: using \texttt{cc\_num} solely as a grouping key for sequence construction lets the LSTM rank a transaction relative to the same card's recent behaviour. Cost-sensitive losses and oversampling, by contrast, should not be applied reflexively in deep sequence models for highly imbalanced data.

\chapter{Introduction}\label{ch:intro}

\section{Background}\label{sec:background}
Credit-card fraud detection is characterised by severe class imbalance: fraudulent transactions account for well below one percent of the data, so naive accuracy is misleading and precision--recall analysis is the only honest evaluation lens \parencite{gunjan_credit_2022}. The economic stakes have grown as digital payments have shifted toward card-not-present authorisation, where verification rests on transactional context rather than physical possession of the card \parencite{xuan_random_2018}, and fraudsters continuously adapt their tactics so that deployed models must operate under behavioural drift rather than under a stationary data-generating process \parencite{afriyie_supervised_2023}.

In our prior study \parencite{Du2026MLFraud}, classical Random Forest and Support Vector Machine classifiers were evaluated on this same dataset. The Random Forest reached fraud-class F\(_1\) of $0.48$ (precision $0.59$, recall $0.41$) on the temporally separated test set, while SMOTE oversampling collapsed it to F\(_1 = 0.13\) and degraded the SVM into a near-trivial classifier. Sequence-aware deep learning is the natural next step: \textcite{mienye_deep_2024} note that recurrent architectures such as LSTMs and GRUs target exactly the temporal-dependency structure of transaction streams, and \textcite{zhou_deep_2025} report consistent gains for fraud detection when models can carry both short- and longer-term information. Whether an LSTM can in turn make explicit imbalance handling unnecessary is the open question motivating the present paper.

\section{Research Question}\label{sec:research_question}
This study examines whether a sequence-based deep learning model can improve fraud detection on the same dataset used in \textcite{Du2026MLFraud}, and whether explicit class-imbalance handling — through cost-sensitive loss reweighting or synthetic minority oversampling — adds value once the model is allowed to condition on per-card transaction history.

The primary research question is:

\textit{Can an LSTM-based model improve fraud detection on this dataset relative to our earlier classical benchmarks, and how do three handling strategies for the severe class imbalance — plain binary cross-entropy, cost-sensitive cross-entropy with} \texttt{pos\_weight} $\approx 172$\textit{, and sequence-level SMOTE — compare in out-of-sample performance?}

Performance is assessed on the temporally separated test set with PR-AUC as the primary ranking metric, supplemented by precision, recall, and F\(_1\) at both the default and F\(_2\)-tuned decision thresholds.

\section{Dataset and Prior-Study Reference}
\label{sec:dataset}

The analysis uses the public credit-card fraud dataset from \textcite{kaggle_fraud_detection_dataset}, distributed as two temporally disjoint files \texttt{fraudTrain.csv} ($1{,}296{,}675$ rows, used for development) and \texttt{fraudTest.csv} ($555{,}719$ rows, held out for final evaluation), corresponding to an approximate 70/30 chronological split. Each row is a single transaction with a binary \texttt{is\_fraud} label; the development file spans early 2019 through mid-2020 and the test file mid-2020 through early 2021. Fraudulent transactions account for roughly $0.6$~percent of the development data and $0.4$~percent of the test data — pronounced class imbalance that drives every modelling decision below. The schema includes 23 transactional, demographic, and geographical fields shared across both files. Full descriptive statistics, the feature schema, and exploratory analysis appear in \textcite{Du2026MLFraud} and Appendix~\ref{app:eda}.

\section{Exploratory Data Analysis}
\label{sec:eda}

Two features of the data shape the modelling choices and are reproduced here for context. First, fraud is rare in the absolute and clearly non-uniform across the population (Figure~\ref{fig:eda_class_imbalance}); evaluation must therefore use precision--recall metrics rather than accuracy, and stratification or cross-split history must be used wherever data is partitioned. Second, fraud rates exhibit pronounced hour-of-day and weekday structure (Figure~\ref{fig:eda_fraud_by_hour}), with elevated incidence in late-night and early-morning windows. This temporal heterogeneity is the most direct empirical motivation for a sequence-aware model: a transaction that is unremarkable in isolation may be anomalous given the time-of-day pattern of the same card's recent activity. Right-skewed transaction amounts, near-zero linear correlations among features, and merchant-category-specific risk are visualised in Appendix~\ref{app:eda}.

\section{Feature Engineering and Preprocessing}
\label{sec:feature_engineering}

Each transaction is transformed into a 78-dimensional numerical feature vector. Calendar fields are split into hour, day-of-week, month, and a weekend indicator; because each is cyclical rather than linear, they are passed through sine/cosine pairs of period $T$,
\[
\sin\!\left(\frac{2\pi t}{T}\right), \qquad \cos\!\left(\frac{2\pi t}{T}\right),
\]
so that 23:00 and 00:00 are treated as neighbours rather than as opposite ends of a numeric scale. Customer age is derived from date of birth, after which raw timestamps and birth dates are dropped. Categorical variables are one-hot encoded for low cardinalities and target-encoded for high-cardinality fields (merchant, city, job, ZIP), with the encoder fit only on the training subset to prevent leakage. Numerical features are standardised under the same train-only fit. Identifier-style variables — credit-card numbers, transaction IDs, names, street addresses — are excluded from the feature set on leakage-prevention grounds, following the rationale of \textcite{Du2026MLFraud} reproduced in Appendix~\ref{app:fe}.

\section{Data Splitting and Leakage Prevention}
\label{sec:data_split}

The development file \texttt{fraudTrain.csv} is partitioned into training and validation subsets via a stratified 80/20 split on the \texttt{is\_fraud} label with a fixed seed of 42; stratification preserves the class ratio in both subsets and ensures that the validation distribution remains representative of the development data. The 80-percent training subset fits both the model and every preprocessing transformation (scaler, one-hot encoder, target encoder); the 20-percent validation subset is used for hyperparameter selection, the imbalance-variant comparison, and threshold tuning. The temporally separated file \texttt{fraudTest.csv} is reserved for a single end-of-pipeline evaluation and never participates in any selection step. The full preprocessing-within-split protocol and the model-selection-versus-final-evaluation distinction are detailed in Appendix~\ref{app:fe}.

\section{Sequence Construction}
\label{sec:sequence_construction}

After preprocessing, transactions are arranged into per-card sequences. For every \texttt{cc\_num}, transactions are sorted by timestamp; for each transaction $i$ a fixed-length window of the up to $L = 20$ most recent transactions of the same card (inclusive of $i$) is constructed, with label given by the \texttt{is\_fraud} indicator of $i$. Preceding labels are not features. Crucially, \texttt{cc\_num} is used only as a grouping key and never enters the feature tensor: the model conditions each prediction on the same card's recent behaviour without memorising cardholder identity. Applied to the temporal split above, the procedure yields $1{,}102{,}173$ training and $194{,}502$ validation sequences ($6{,}373$ and $1{,}133$ fraudulent) and $555{,}719$ test sequences ($2{,}145$ fraudulent).

Two implementation details shape what the LSTM sees inside each window. Padded positions are not processed by the recurrent cell: right-padded windows are passed through \texttt{pack\_padded\_sequence}, the cell state updates only on real transactions, and the hidden state read by the classification head is the one produced after the last valid timestep, recovered via a per-row gather. When constructing validation and test windows we additionally prepend the last $L - 1$ transactions per card from the earlier split(s) as non-anchor history rows, so a transaction near a split boundary is not artificially short-handed. Full mechanics are in Appendix~\ref{app:seq}.

\section{LSTM Architecture and Training}
\label{sec:lstm}

LSTM networks are recurrent neural networks designed to carry information over long sequences via a gated cell state \parencite{olah_understanding_2015}; this is well suited to fraud detection because suspicious activity may emerge from a transaction's position within a card's history rather than from its features in isolation. The cell equations are reproduced in Appendix~\ref{app:lstm_eq} for reference.

The model has three components. An input projection maps each 78-dimensional feature vector to 64 dimensions, decoupling the input dimension from the recurrent hidden size and providing a small amount of learned linear preprocessing. Two stacked unidirectional LSTM layers with hidden size 128 process the projected sequence; the depth is sufficient to learn compositional temporal patterns over windows of length 20 without the cost of a deeper stack. A small classification head applies dropout, projects from 128 to 64 dimensions through a ReLU non-linearity, and emits a single scalar logit; a sigmoid is applied at evaluation time to obtain $\hat{p}(y = 1 \mid x_{1:T})$.

Optimisation uses AdamW with learning rate $3\times 10^{-4}$, weight decay $10^{-5}$, and dropout $0.3$, both selected from the validation grid in Section~\ref{subsec:lstm_hpt}. Training runs for at most 30 epochs at batch size 128. Early stopping monitors validation PR-AUC with patience 5: the checkpoint with the highest observed validation PR-AUC is retained. After training, a single decision threshold is chosen on the validation set by maximising F\(_2\), which weights recall four times more than precision and reflects the operational asymmetry of fraud detection. The selected threshold is frozen and applied unchanged to the test set; test results are reported at both the default $\tau = 0.5$ and the F\(_2\)-tuned threshold so that any contribution of threshold tuning to the headline numbers is transparent.

\section{Class-Imbalance Variants}
\label{sec:imbalance}

To examine which interventions, if any, the present setting requires, the LSTM is trained under three imbalance-handling regimes: plain binary cross-entropy on the imbalanced data, cost-sensitive binary cross-entropy with positive-class weight \texttt{pos\_weight} $\approx 172$, and standard binary cross-entropy on a SMOTE-rebalanced training set. Cost-sensitive loss and SMOTE were both evaluated, in different forms, in \textcite{Du2026MLFraud}; the present paper revisits both in the deep-learning setting and adds plain BCE as a no-intervention reference. The cost-sensitive loss rescales the positive-class term in the per-sample BCE,
\begin{equation}
\mathcal{L}(z, y) = -\,w_{+}\, y \log \sigma(z) - (1 - y) \log\!\bigl(1 - \sigma(z)\bigr),
\qquad w_{+} = \frac{N_{\text{neg}}}{N_{\text{pos}}} \approx 172,
\end{equation}
and is implemented as \texttt{BCEWithLogitsLoss} with the \texttt{pos\_weight} argument \parencite{chawla_smote_2002}. Sequence-level SMOTE flattens each window, interpolates within the minority class via \texttt{imbalanced-learn}, and reshapes back to the sequence layout; a $50\%$ training subset is used because the resampled tensor exceeds the host's memory budget (Appendix~\ref{app:smote_mem}).

\section{Hyperparameter Grid}
\label{subsec:lstm_hpt}

The grid varies optimiser learning rate and dropout, two settings that most directly affect optimisation under imbalance and the train/validation gap; remaining hyperparameters (hidden size, number of layers, sequence length, batch size, projection dimension) are held fixed at the defaults motivated above. The full-factorial grid covers $\text{lr} \in \{3\!\times\!10^{-4},\ 10^{-3},\ 3\!\times\!10^{-3}\}$ and $\text{dropout} \in \{0.3,\ 0.5\}$. All six trials share the same seed so observed differences reflect the hyperparameters rather than initialisation noise. Table~\ref{tab:lstm_hpt} reports validation PR-AUC for each cell. The best configuration ($\text{lr} = 3\times 10^{-4}$, $\text{dropout} = 0.3$) reaches $0.987$, but the optimum is broad: five of six cells lie within a $0.003$ band, and only $\text{lr} = 3\times 10^{-3}$ at the higher dropout meaningfully under-performs. The grid is run on the plain-BCE variant and the picked configuration is reused for the cost-sensitive and SMOTE variants. Full configuration rationale is in Appendix~\ref{app:hp_grid}.

\section{Imbalance-Variant Comparison on Validation}
\label{subsec:val_imbalance}

The three imbalance-handling variants of the LSTM share architecture, optimiser, and the picked $(\text{lr}, \text{dropout})$ and differ only in the loss configuration. Each is evaluated on the held-out validation split. Table~\ref{tab:lstm_imbalance_val} reports validation PR-AUC at the best epoch and the F\(_2\)-optimal decision threshold estimated on each variant's validation probabilities. The plain-BCE configuration reaches $0.987$, exceeding the cost-sensitive variant by roughly one percentage point and SMOTE by eleven. The F\(_2\)-tuned thresholds differ sharply: cost-sensitive requires a near-extreme $\tau \approx 0.981$, whereas plain BCE and SMOTE land near $0.028$ and $0.127$. This reflects the large positive-class weight in cost-sensitive training, which inflates positive logits and pushes the score distribution to the right-most edge of the unit interval; plain BCE retains probability calibration. Figure~\ref{fig:lstm_score_distribution} makes this calibration shift visually explicit. The plain-BCE configuration is therefore selected as the baseline LSTM for the test-set comparison; the other two are reported alongside it for transparency.

The headline gain — PR-AUC from $0.509$ to $0.969$ — is unusually large and reflects a structural rather than statistical advantage. The Random Forest scored each transaction in isolation, with \texttt{cc\_num} either dropped or target-encoded into a per-card fraud rate. The LSTM never reintroduces \texttt{cc\_num} as a feature but uses it as a grouping key for sequence construction: each prediction is conditioned on the same card's preceding transactions in time order. The model can flag a transaction as anomalous \emph{relative to that card's recent behaviour}, which a row-wise classifier is architecturally unable to do.

The two imbalance-handled LSTM variants outperform every classical baseline but underperform the plain-BCE configuration. Cost-sensitive (PR-AUC $0.959$, F\(_2\)-tuned $F_1 = 0.89$) loses about one percentage point of PR-AUC and four of F\(_1\); SMOTE (PR-AUC $0.890$, F\(_2\)-tuned $F_1 = 0.83$) loses about eight. The directional finding — that synthetic oversampling does not help under realistic temporal evaluation — reproduces \parencite{Du2026MLFraud}, where SMOTE collapsed Random Forest F\(_1\) from $0.48$ to $0.13$, while attenuating its magnitude in the deep-learning setting.


\pagebreak
\chapter{Discussion}\label{ch:discussion}

\section{Interpretation and Comparison with Prior Work}
\label{sec:interpretation}

Our LSTM-based fraud detector substantially outperforms the classical baselines from the prior project \parencite{Du2026MLFraud}. The plain-BCE configuration achieves PR-AUC $0.969$ on the held-out test set, compared to $0.509$ for the Random Forest and $0.101$ for the SVM. The fraud-class F\(_1\) score nearly doubles, from $0.48$ to $0.94$ at the default threshold; ROC-AUC moves from $0.982$ to $0.998$, indicating that the LSTM ranks fraudulent transactions almost perfectly above legitimate ones. The false-positive count at the default threshold drops to $52$ across the $553{,}574$ legitimate test transactions — an operationally meaningful number when each false positive translates into an analyst review or a customer interruption.

We attribute this gap primarily to a \emph{structural} change rather than to additional model capacity. Both pipelines see the same underlying transaction features. The decisive difference is how \texttt{cc\_num} is used. The Random Forest treated each transaction in isolation, with \texttt{cc\_num} either dropped (to avoid memorisation) or target-encoded into a per-card fraud prevalence, neither of which generalises to unseen cardholders. The LSTM uses \texttt{cc\_num} only as a grouping key to construct per-card transaction sequences, while the identifier itself never enters the feature tensor. The model therefore learns shapes of behaviour over a card's recent history. A 400 USD charge at 03:14 looks identical to a row-wise classifier; a sequence model can recognise it as anomalous if the card's prior nineteen transactions were small daytime purchases.

\section{Why Imbalance Handling Hurts Here}
\label{sec:imbalance_discussion}

The three variants produce a clear ordering: plain BCE ($0.969$) > cost-sensitive ($0.959$) > SMOTE ($0.890$). This may appear counterintuitive given the severe class imbalance but reflects three observations. First, plain BCE remains well-calibrated, with predicted probabilities tracking observed fraud frequencies, while a positive-class weight of $172$ inflates positive logits and produces a strongly bimodal score distribution that the F\(_2\) tuner can only correct by selecting a threshold near $0.98$. Second, sequence-level SMOTE interpolates between flattened windows of length 20, and the resulting synthetic histories are unlikely to lie on the manifold of plausible per-card behaviour: they inject noise rather than informative minority signal. Third, with $6{,}373$ fraud sequences in the training subset, the LSTM already has enough positive examples to learn discriminative shapes; explicit rebalancing supplies no marginal information.

\section{Limitations}
\label{sec:limitations}

\paragraph{Single random seed.} All training runs and the grid use a fixed seed of $42$. This isolates hyperparameter effects from initialisation noise but prevents an estimate of run-to-run variance; the small differences between adjacent grid cells should therefore not be over-interpreted.

\paragraph{Hyperparameter scope and selection on a single variant.} The grid covers only learning rate and dropout, with three and two values respectively, and is run on the plain-BCE variant; the picked $(\text{lr}, \text{dropout})$ is then reused for the cost-sensitive and SMOTE variants. A broader and per-variant search could plausibly close part of the gap reported above (Appendix~\ref{app:limitations}).

\paragraph{Sequence-level SMOTE memory budget.} The flattened-window SMOTE step in \texttt{imblearn} peaks at roughly $14$~GB on the full training set and forces SMOTE to use a $50\%$ subset; the comparison with the other two variants is therefore not strictly apples-to-apples (Appendix~\ref{app:smote_mem}).

\paragraph{Synthetic data.} The Sparkov simulator is not a real-world distribution, and fraud signatures in deployed systems evolve in response to defences. Generalisation claims should be limited to data of the type studied here.

\section{Future Work and Conclusion}
\label{sec:conclusion}

The plain-BCE LSTM closes most of the gap left by the classical baselines, achieving PR-AUC $0.969$ versus $0.509$ for the Random Forest, with explicit imbalance handling actively hurting rather than helping in this regime. Future work could test attention or transformer encoders for longer-range dependencies, explicitly model temporal drift between training and deployment, run multi-seed and per-variant grids, and validate on real (non-synthetic) transaction data to test whether the structural advantage of per-card sequence modelling carries over to adversarially adapted fraud distributions.

The training dataset contains approximately 1.3 million individual credit-card transactions recorded between early 2019 and mid-2020. \section{Feature Engineering Details and Leakage Prevention}
\label{app:fe}

Several identifier-like variables — credit-card numbers, transaction identifiers, names, street-level addresses, and raw timestamps — are excluded from the feature set. Although these may appear highly predictive within the observed sample, they primarily allow the model to memorise specific entities or historical incidents rather than learn transferable behavioural patterns. \textcite{kaufman_leakage_2012} demonstrate that high-cardinality identifiers often exhibit strong apparent predictive power due to data-collection artifacts and sampling bias, leading to overly optimistic performance estimates. In the context of fraud detection, \textcite{dal_pozzolo_credit_2015} emphasise that repeated entity occurrences can bias evaluation and hinder generalisation under temporal and behavioural drift. Identifier-based variables are therefore removed in order to reduce leakage risk and to ensure that the model is evaluated on its ability to detect fraud-related behaviour rather than to recall specific entities from the historical sample.

The preprocessing pipeline applied within the development split comprises:
\begin{itemize}
    \item Standardisation of numerical features using \texttt{StandardScaler},
    \item One-hot encoding for low-cardinality categorical variables using \texttt{OneHotEncoder},
    \item Target encoding for high-cardinality categorical variables using \texttt{TargetEncoder}.
\end{itemize}
These transformations are implemented within a single pipeline object such that \texttt{fit} is applied only on $(X_{\text{train}}, y_{\text{train}})$ and the resulting transformations are then applied unchanged to $X_{\text{val}}$ and $X_{\text{test}}$. Scaling parameters, one-hot categories, and target-encoding mappings are estimated solely from the training subset.

\paragraph{Model selection versus final evaluation.} The internal validation set is used for model selection. For the LSTM, hyperparameter selection focuses on the optimiser learning rate and the dropout rate; remaining structural hyperparameters (hidden size, number of layers, projection dimension, sequence length, batch size) are held fixed at the values motivated in the body. Selection is based on validation PR-AUC. After the best configuration is selected, results are reported in two places: internal validation performance reflects model-selection quality within the development sample, and external test performance reflects final generalisation to unseen and temporally separated data.


\section{Sequence Construction Details}
\label{app:seq}

For every credit-card identifier \texttt{cc\_num}, the corresponding transactions are sorted in ascending order of \texttt{trans\_date\_trans\_time}. For each transaction $i$ of card $c$, a fixed-length window is constructed from the up to $L = 20$ most recent transactions of the same card with a timestamp less than or equal to that of transaction $i$, inclusive. The label associated with the window is the \texttt{is\_fraud} indicator of transaction $i$; the labels of the preceding transactions in the window are not used as features. When fewer than $L$ prior transactions are available for the same card, the window is right-padded with zero vectors and the original number of valid transactions is recorded as a length mask.

\texttt{cc\_num} is used only as a grouping key for sequence construction and is not included in the feature set; the card identity therefore never enters the model tensor. What the model sees is the same row-level feature representation as the classical baselines, with the additional structure that each prediction is conditioned on the recent transaction history of the same card. This distinction allows the sequence model to capture per-card behavioural context without re-introducing the entity-memorisation risk associated with high-cardinality identifiers.

\paragraph{Right-padded packing.} Right-padded windows are passed to the LSTM via \texttt{pack\_padded\_sequence}, so the cell state is updated only by real transactions; the hidden state read by the classification head is the one produced after the last valid timestep, recovered via a per-row gather. This avoids a subtle bias in which short-history windows would otherwise be dominated by zero-feature padding rows that, after standardisation, look like ``mean'' transactions to the model.

\paragraph{Cross-split history.} When constructing validation and test sequences, the last $L - 1$ transactions per card from the earlier split(s) are prepended as non-anchor history rows: the windowing logic uses them to fill earlier window positions but does not emit a window centred on them. As a result, the first transaction of a card in the validation slice no longer has length one purely because the split boundary truncates its history; it can use up to nineteen prior transactions from the training period instead. Test windows use train and validation history in the same way. This change does not alter the number of emitted windows, only the contents of windows that span a split boundary.


\section{LSTM Cell Equations and Gating Walkthrough}
\label{app:lstm_eq}

Let $x_1, x_2, \dots, x_T$ denote a sequence of transaction feature vectors. At each time step $t$, the LSTM receives the current input $x_t$, the previous hidden state $h_{t-1}$, and the previous cell state $C_{t-1}$. The model then computes
\[
f_t = \sigma \left( W_f [h_{t-1}, x_t] + b_f \right), \quad \text{(forget gate)}
\]
\[
i_t = \sigma \left( W_i [h_{t-1}, x_t] + b_i \right), \quad \text{(input gate)}
\]
\[
\tilde{C}_t = \tanh \left( W_C [h_{t-1}, x_t] + b_C \right), \quad \text{(candidate cell state)}
\]
\[
C_t = f_t \odot C_{t-1} + i_t \odot \tilde{C}_t, \quad \text{(cell state update)}
\]
\[
o_t = \sigma \left( W_o [h_{t-1}, x_t] + b_o \right), \quad \text{(output gate)}
\]
\[
h_t = o_t \odot \tanh(C_t), \quad \text{(hidden state)}
\]

The forget gate determines which parts of the old memory are retained. The input gate determines which new information enters the memory. The output gate determines which parts of the updated memory are exposed to the hidden state. In contrast to a standard RNN, where the hidden state is fully overwritten at each step, the cell state is updated through a gated additive rule. Old information is retained or discarded by the forget gate, and new information is added through the input gate. This gives the model a more stable way to carry information through time and helps it learn longer-range dependencies \parencite{olah_understanding_2015}.


\section{Imbalance-Handling Mathematical Specification}
\label{app:imbalance_math}

\paragraph{Cost-sensitive loss.} For a logit $z$ and a binary label $y \in \{0, 1\}$, the per-sample cost-sensitive loss is
\[
\mathcal{L}(z, y) = -\,w_{+}\, y \log \sigma(z) - (1 - y) \log\!\bigl(1 - \sigma(z)\bigr),
\]
where $\sigma(\cdot)$ denotes the logistic function and $w_{+}$ is a positive-class weight. The weight is set to the empirical class ratio $w_{+} = N_{\text{neg}} / N_{\text{pos}}$, which makes the contribution of the minority class to the loss approximately equal to that of the majority class. On the training subset this evaluates to $w_{+} \approx 172$. The formulation is implemented as PyTorch's \texttt{BCEWithLogitsLoss} with the \texttt{pos\_weight} argument and is the deep-learning analogue of class-weight adjustments commonly used for tree-based and linear classifiers.

\paragraph{Sequence-level SMOTE.} For each minority-class observation $\mathbf{x}_i \in \mathbb{R}^d$, $k = 5$ nearest neighbours are identified within the minority class using Euclidean distance \parencite{chawla_smote_2002}. Synthetic samples are generated by selecting a neighbour $\mathbf{x}_{\text{nn}}$ at random and computing
\[
\mathbf{x}_{\text{synthetic}} = \mathbf{x}_i + \lambda \left( \mathbf{x}_{\text{nn}} - \mathbf{x}_i \right),
\]
with $\lambda \sim \mathcal{U}(0,1)$. Because the LSTM operates on per-card sliding windows rather than on individual transactions, SMOTE is applied at the sequence level: each window is flattened into a single high-dimensional vector, the interpolation is run in that flattened space, and the resulting synthetic vectors are reshaped back into the original sequence layout. The minority class is up-sampled to ten percent of the majority class. Synthetic samples are generated only from the training portion of the development sample to avoid leakage; validation and test data preserve the original class distribution. When SMOTE is used, the positive-class weight in the loss is set to one so that the loss reduces to standard binary cross-entropy on a now-balanced training set and the two imbalance-handling mechanisms do not compound.


\section{Hyperparameter Grid Configuration Notes}
\label{app:hp_grid}

Within the broader hyperparameter inventory of the model — number of recurrent layers, hidden size, dropout, learning rate, batch size, and sequence length — two are varied empirically and the remainder are held fixed. The two varied hyperparameters are the optimiser learning rate and the dropout rate applied between the recurrent layers and within the classification head: the learning rate governs the optimisation regime under the large positive-class weight, while dropout is the most direct lever on the train/validation gap that motivates regularisation in the first place. The two dropout values bracket a moderate and a stronger regularisation regime, while the three learning-rate values are spaced geometrically and bracket both more conservative and more aggressive optimisation regimes around a typical default of $1\times 10^{-3}$.

Every trial is trained from the same initialisation seed, so that observed differences across trials reflect the varied hyperparameters rather than initialisation noise. An explicit estimate of run-to-run variance is therefore beyond the scope of this experiment. Each trial is otherwise identical to the baseline configuration: the same training, validation, and test sequences, the same loss function and optimiser, the same early-stopping protocol with patience five, and the same F\(_2\)-based threshold-selection step on the validation set. The grid is run on the plain-BCE variant — ultimately selected as the baseline LSTM in the body — so that the chosen $(\text{lr}, \text{dropout})$ is optimised directly on the model class that drives the comparison rather than on a different loss configuration.


\section{SMOTE Memory Footprint and 50\% Subset Caveat}
\label{app:smote_mem}

The SMOTE variant uses a $50\%$ subset of the training set rather than the full $\sim 1.1$ million sequences. The flattened-window SMOTE step in \texttt{imblearn} returns originals and synthetic samples in a single resampled tensor, which on the full set requires roughly two times the train-tensor footprint at peak ($\approx 14$~GB) and exceeds the available memory of the $16$~GB training host. The cost-sensitive and plain-BCE variants do not have this memory overhead and use the full training set. This asymmetry is acknowledged here; the resulting validation and test numbers for the SMOTE variant are therefore computed from a smaller training pool and should be interpreted with this caveat in mind.


\section{Extended Limitations}
\label{app:limitations}

\paragraph{Threshold tuning and operational asymmetry.} F\(_2\)-based threshold tuning on the validation set materially shifts the operating point of all three variants toward higher recall, consistent with the operational asymmetry of fraud detection in which a missed fraudulent transaction is more costly than a false-positive review. The tuned thresholds vary widely across variants ($0.028$ for plain BCE, $0.981$ for the cost-sensitive variant, $0.127$ for SMOTE), reflecting the different probability distributions induced by each imbalance-handling strategy rather than a difference in underlying model quality. The very small plain-BCE threshold is a property of a well-calibrated score distribution: legitimate transactions cluster tightly near zero, while a residual tail of low-confidence fraud sits in the $[0.03, 0.5)$ range and is captured by lowering the threshold without flagging large numbers of additional false positives.

\paragraph{Hyperparameter selection on a single variant.} The grid is run on the plain-BCE variant and the resulting $(\text{lr}, \text{dropout})$ is reused for the cost-sensitive and SMOTE variants. This matches the paper's narrative — tune the simplest training setup, then ask whether adding imbalance handling improves on it — but it implicitly assumes that the optimal hyperparameters do not differ substantially across loss configurations. A separate grid for each variant could in principle yield different picked values, particularly for SMOTE which trains on a rebalanced distribution.

\paragraph{Hyperparameter grid scope.} The grid search covers only learning rate and dropout, with three and two values respectively. Other hyperparameters such as hidden size, sequence length, and number of recurrent layers were held fixed at defensible defaults but were not optimised. It is plausible that a broader search would yield modest further gains, although the four most central cells of the grid lie within a half-percentage-point band of validation PR-AUC, indicating that the optimum is broad rather than sharp.
